# 多模态图文智能分析助手 正式方案文档

## 1. 项目概述

### 1.1 项目名称

多模态图文智能分析助手

### 1.2 项目背景

在实际办公、学习和资料整理场景中，大量文档以图文混合形式存在，例如产品说明书、研究报告、课件截图、合同扫描件、技术文档和图表资料等。这类文档通常同时包含自然语言文本、图片、图表、流程图及扫描文字，人工阅读和整理成本较高。

随着多模态大模型和OCR技术的发展，系统已经具备对图像与文本联合理解的能力。因此，设计并实现一个轻量化、本地部署友好的Web应用，用于自动提取、理解和总结图文混合文档内容，并支持基于文档内容的问答交互，具有较强的实用价值和应用前景。

### 1.3 项目目标

本项目拟开发一个在本地尽可能轻量的Web应用。用户上传包含图片和文本的混合文档后，系统能够自动完成以下任务：

1. 提取并理解文档中的图像和文本内容。
2. 将文档转换为统一的结构化中间表示。
3. 生成标准化的Markdown总结报告。
4. 支持基于文档内容的自然语言问答。

系统采用本地文档解析与OCR、云端大模型API推理相结合的模式，在保证功能完整性的同时兼顾实现复杂度、部署成本和扩展能力。

## 2. 建设范围

### 2.1 建设内容

本系统面向单用户、本地运行场景，围绕“上传文档 - 自动分析 - 生成报告 - 文档问答”的主流程进行设计与实现，重点完成以下能力：

1. 多格式文件上传与解析。
2. 图文混合内容提取与理解。
3. 结构化总结报告生成。
4. 基于检索增强的单轮问答。
5. 分析结果在线预览与Markdown导出。

### 2.2 非建设内容

为控制项目范围、确保MVP可落地，以下内容不纳入本阶段实现：

1. 不支持视频、音频等非静态文档输入。
2. 不进行精细表格结构还原，表格内容按普通文本处理。
3. 不实现多轮对话记忆，每次问答彼此独立。
4. 不提供用户注册、登录、权限管理等账户系统。
5. 不做长期文件加密存储，文件仅用于临时处理。

## 3. 用户与使用场景

### 3.1 目标用户

1. 需要快速阅读和整理资料的学生与教师。
2. 需要处理说明书、报告和图表资料的办公人员。
3. 需要对扫描件、截图和图文混合文档进行快速分析的普通用户。

### 3.2 典型使用场景

1. 用户上传一份图文混合PDF，例如产品说明书或研究报告。
2. 用户上传包含文字说明和截图的Word文档。
3. 用户上传一张或多张图片，例如网页截图、实验图表或拍照扫描页。
4. 用户在系统生成总结后，继续提问“图2中曲线表示什么”或“产品最大功率是多少”等问题。

## 4. 系统功能需求

### 4.1 功能一：多格式文档上传

系统应支持以下输入格式：

1. PDF文档，包括电子版PDF和扫描版PDF。
2. Word文档，仅支持`.docx`格式。
3. 纯图片文件，包括`.jpg`、`.jpeg`、`.png`。

上传约束如下：

1. 单次最多上传5个文件。
2. 上传文件总大小不超过20MB。
3. 支持单文件分析与多文件批量上传。

### 4.2 功能二：图文内容提取与理解

系统在接收到用户文件后，应执行统一解析流程，将不同类型文档转换为标准化内容块。

#### 4.2.1 文本提取

1. 对电子版PDF，直接提取可读文本段落。
2. 对Word文档，提取正文段落、标题及可读取的文本内容。
3. 对扫描页或图片中的文字区域，使用OCR提取文本。

#### 4.2.2 图片提取与理解

1. 从PDF和Word中提取嵌入图片。
2. 对纯图片文件直接作为图像输入处理。
3. 若图片中包含较多文字内容，优先采用OCR提取。
4. 若图片为图表、照片、示意图、流程图等非纯文字图片，调用多模态大模型生成简要描述。
5. 当图像理解失败时，系统应标注“无法识别”，但不能中断整体流程。

#### 4.2.3 中间结构表示

为统一后续总结与问答逻辑，系统内部应将解析结果组织为统一结构，例如：

```json
[
  {
    "type": "text",
    "file_name": "example.pdf",
    "page": 1,
    "content": "正文段落内容"
  },
  {
    "type": "image",
    "file_name": "example.pdf",
    "page": 2,
    "image_index": 1,
    "ocr_text": "图片中的文字",
    "description": "图片内容描述"
  }
]
```

在此基础上，系统进一步生成带标注的纯文本表示，用于报告生成与检索问答。

### 4.3 功能三：总结报告生成

系统应基于已提取的文本内容和图片描述，自动生成Markdown格式总结报告。报告结构必须至少包含以下内容：

```markdown
# 文档总结报告
## 1. 核心摘要
## 2. 关键信息提取
- 主题/领域
- 主要结论
- 数据/指标
## 3. 图表/图片说明
- 图片1：[描述]
- 图片2：[描述]
## 4. 待确认或疑问点
```

具体要求如下：

1. 核心摘要控制在150字以内。
2. 应尽可能提取关键数字、指标、结论和主题。
3. 图像描述应与原图片编号或页码建立对应关系。
4. 报告生成后支持在线预览。
5. 报告支持保存为`.md`文件并下载。

### 4.4 功能四：基于文档内容的问答交互

系统应提供自然语言问答能力，用户可围绕当前已上传文档进行单轮提问。

示例问题包括但不限于：

1. 图2中曲线代表什么？
2. 文档的主要结论是什么？
3. 产品最大功率是多少？
4. 第3页图片中的设备用途是什么？

问答功能要求如下：

1. 问答仅基于当前已解析文档内容进行回答。
2. 系统优先从结构化文本和图片描述中检索相关片段。
3. 回答应尽量简洁明确。
4. 在必要时，回答中应引用原文片段或图片描述。
5. 若支持来源展示，应标注文件名、页码或图片编号。

## 5. 技术方案设计

### 5.1 总体设计原则

本项目遵循以下设计原则：

1. 轻量优先：本地侧仅负责文档解析、OCR和基础检索，复杂推理交由云端API完成。
2. 模块解耦：文档解析、OCR、模型调用、检索、问答和前端展示相互独立，便于后续替换。
3. MVP优先：优先实现核心链路，避免过早引入复杂的工程组件。
4. 可扩展：为后续接入其他多模态模型或本地模型预留统一接口。

### 5.2 推荐技术选型

| 层次 | 技术选型 | 说明 |
| --- | --- | --- |
| 前端界面 | Streamlit | 快速实现文件上传、报告展示和问答交互 |
| 服务逻辑 | Python | 实现文档解析、流程编排和结果组织 |
| PDF解析 | PyMuPDF | 提取文本、页面和嵌入图片 |
| Word解析 | python-docx | 提取段落和图片资源 |
| OCR | PaddleOCR | 本地提取图片文字 |
| 多模态理解 | 云端MLLM API | 用于图像描述和图文理解 |
| 文本大模型 | 云端LLM API | 用于总结生成与问答生成 |
| 检索模块 | FAISS或轻量向量检索 | 用于问答相关片段召回 |
| 文件管理 | 本地临时目录 | 用于缓存上传文件和中间产物 |

说明：

1. 若采用Streamlit实现MVP，可暂不单独拆分FastAPI后端。
2. 若后期需要前后端分离、接口复用或多人协作，再引入FastAPI更为合适。

### 5.3 模块划分

系统可划分为以下模块：

1. 文件上传模块：负责接收PDF、DOCX、图片文件并进行基础校验。
2. 文档解析模块：负责抽取文本、图片、页码和结构信息。
3. OCR模块：负责识别图像中的文字内容。
4. 多模态理解模块：负责对图表、流程图和照片类图片生成描述。
5. 内容融合模块：负责将文本块与图片块合成为统一中间文档。
6. 总结生成模块：负责根据融合文档生成Markdown总结报告。
7. 检索问答模块：负责对文档内容建立索引并完成问题召回与回答。
8. 前端展示模块：负责展示报告、问答结果和来源信息。

## 6. 核心业务流程

### 6.1 主流程

系统主流程如下：

1. 用户上传一个或多个文档。
2. 系统校验文件格式、数量和大小。
3. 系统按文件类型执行解析：
   - PDF提取文本与图片。
   - DOCX提取段落与图片。
   - 图片文件直接进入图像处理流程。
4. 对图片执行内容判断：
   - 含文字图片优先OCR。
   - 图表、流程图、照片等送多模态模型描述。
5. 系统生成统一的图文混合中间文档。
6. 调用文本大模型生成Markdown总结报告。
7. 对融合后的文本块建立检索索引。
8. 用户提出问题后，系统先检索相关内容，再由大模型生成答案。

### 6.2 问答流程

1. 接收用户问题。
2. 将问题编码并在文档索引中检索相关片段。
3. 选取若干最相关内容作为上下文。
4. 调用文本大模型生成简洁答案。
5. 返回答案及对应来源位置。

## 7. 非功能性需求

### 7.1 性能要求

1. 单个常规文档的解析与总结生成目标时间不超过30秒，不含云端模型排队时间。
2. 多文件场景下，总处理时长允许随页数和图片数量线性增长。
3. 界面应提供处理中状态提示，避免用户误判系统无响应。

### 7.2 成本控制要求

1. 单次分析平均多模态模型调用次数不超过5次。
2. 优先采用“先OCR，后多模态补充描述”的策略，避免每张图均调用MLLM。
3. 文本总结与问答尽量复用已有结构化结果，减少重复调用。

### 7.3 稳定性要求

1. 当单张图片OCR失败或图像描述失败时，系统应继续处理其他内容。
2. 发生异常时应给出可理解的错误提示，例如“文件格式不支持”或“图片无法识别”。
3. 临时文件处理完成后应自动清理。

### 7.4 可扩展性要求

1. 预留统一模型接口，便于未来替换Qwen-VL、GPT-4V或本地模型。
2. 检索模块应支持后续切换不同向量化模型和向量存储方式。
3. 未来可扩展来源高亮、缩略图展示和跨文档联合分析能力。

## 8. 数据与中间结果设计

### 8.1 文档内容块设计

系统建议将每个文档拆分为内容块，核心字段包括：

1. 文件名
2. 文件类型
3. 页码或位置
4. 内容块类型
5. 文本内容
6. OCR结果
7. 图片描述
8. 图片编号
9. 前后文片段

### 8.2 来源追踪设计

为支持“图2”“第3页”等定位型问题，系统应在解析阶段保留以下信息：

1. 文件名
2. 页码
3. 图片序号
4. 图片前后的相邻文本
5. 原始块ID

该设计有助于后续实现答案引用、高亮来源和调试排障。

## 9. 交付物与验收标准

### 9.1 交付物清单

项目最终交付应至少包括：

1. 可运行的Web应用程序。
2. 启动命令与简明运行说明。
3. 支持上传并分析以下至少三类示例文件：
   - 图文混合PDF
   - 带文字的图片
   - 纯文本加表格的Word文档
4. 一份自动生成的示例Markdown报告。
5. 五个典型问答测试样例及预期答案。
6. 完整代码仓库及README说明文档。

### 9.2 验收标准

项目验收时，应至少满足以下条件：

1. 系统能够成功上传并解析PDF、DOCX和图片三类文件。
2. 系统能够从文档中提取文本内容和图片信息。
3. 系统能够生成符合固定结构的Markdown总结报告。
4. 系统能够根据文档内容回答用户提出的基本问题。
5. 系统在图片识别失败时不会整体中断。
6. 系统界面可完成上传、分析、报告预览和问答操作。

## 10. 风险分析与应对措施

### 10.1 主要风险

1. 扫描版PDF文本质量差，OCR识别准确率可能不足。
2. 图表、流程图等复杂图片的多模态描述可能不稳定。
3. 多文件批量上传时，处理耗时可能超过预期。
4. 云端模型接口可能受到网络或调用额度限制。

### 10.2 应对措施

1. 对OCR结果允许保留原样，并在报告中标注可能存在识别误差。
2. 对图片采用分层策略，优先OCR，必要时再调用多模态模型。
3. 对上传数量、文件大小和图片处理次数设置上限。
4. 对模型调用过程增加异常捕获和降级逻辑。

## 11. 后续优化方向

在完成MVP后，系统可逐步扩展以下能力：

1. 报告中展示图片缩略图及对应说明。
2. 问答结果高亮引用来源，例如文件名、页码和图片编号。
3. 支持多文档联合问答与对比分析。
4. 引入FastAPI实现前后端分离架构。
5. 尝试接入本地轻量模型以降低云端调用成本。

## 12. 结论

本项目聚焦于图文混合文档的自动解析、理解、总结和问答，功能定位明确，边界清晰，具备较强的实用性和可实现性。通过“本地轻量处理 + 云端模型推理”的技术路线，既能控制系统复杂度，又能充分利用当前多模态模型的能力，适合作为课程设计、毕业设计或轻量原型系统进行开发与展示。

从MVP角度看，本方案具有良好的落地条件。只要在实现过程中坚持统一中间结构、控制模型调用成本、优先完成核心主链路，即可较稳定地交付一个可运行、可演示、可扩展的多模态图文智能分析助手。
