# 多模态图文智能分析助手：系统流程说明文档

## 1. 系统定位

本项目是一个本地轻量 Web 应用，用于处理 PDF、DOCX、图片等图文混合文档，完成文档解析、图片理解、结构化总结、RAG 问答和 Agent 工具调用。

当前系统已经体现：

- MLLM：通过 Qwen-VL 系列模型对图片、截图、图表、流程图进行语义描述。
- OCR：通过本地 OCR 提取图片中文字，作为多模态理解的补充。
- LangChain：使用 Document、Retriever、Retrieval Chain、StructuredTool、tool-calling Agent 等组件。
- RAG：将文本、图片描述、OCR、表格内容统一转为可检索内容块，基于 BM25 + FAISS + embedding 召回。
- Agent：将 OCR、图片分析、文本检索、报告摘要等能力封装为工具，并支持工具调用失败时的确定性兜底计划。
- MCP：提供本地 MCP-style Server/Client，支持资源读取和工具调用，并统一工具响应结构。
- 状态管理：使用 SQLite 保存用户主动保存的历史会话、文档块、报告、问答和 Agent 轨迹。

## 2. 总体架构

```text
用户
  -> Streamlit 前端
  -> 文件保存服务
  -> 文档解析层
      -> PDF 解析
      -> DOCX 解析
      -> 图片解析
  -> OCR + Qwen-VL 图片理解
  -> 统一内容块 parsed_documents
  -> Markdown 报告生成
  -> 页面预览 / 下载
  -> 用户手动保存历史
      -> SQLite 持久化
  -> 问答交互
      -> 问题路由
      -> RAG 或 Agent
      -> 答案校验
      -> 来源展示
```

系统目前仍是单体式 Python 应用，模块之间主要通过函数调用通信；MCP 作为本地工具与资源访问层，为后续接入更标准的 Agent 生态预留接口。

## 4. 文档分析流程

### 4.1 上传与临时保存

入口文件：`app.py`

用户上传 PDF、DOCX、PNG、JPG 文件后，系统先校验数量和大小，再调用：

```python
saved_paths = save_uploaded_files(uploaded_files, session_temp_dir)
```

注意：当前版本已经改为“历史会话手动保存”。点击“开始分析”后，文件和报告只会写入 `output/temp/unsaved-*` 与 `output/reports/unsaved-*`，不会自动写入 SQLite 历史。

### 4.2 文档解析

入口文件：`services/parser_service.py`

系统根据文件后缀分发到不同解析器：

```text
PDF   -> parsers/pdf_parser.py
DOCX  -> parsers/docx_parser.py
Image -> parsers/image_parser.py
```

所有解析器最终输出统一结构：

```json
{
  "file_name": "demo.pdf",
  "file_type": "pdf",
  "blocks": [
    {
      "type": "text",
      "page": 1,
      "content": "正文内容",
      "source": "demo.pdf 第 1 页"
    },
    {
      "type": "image",
      "page": 2,
      "image_index": 1,
      "image_path": "本地图片路径",
      "ocr_text": "图片中的文字",
      "description": "图片语义描述",
      "content": "图片描述 + OCR 文本",
      "source": "demo.pdf 第 2 页 图片 1"
    },
    {
      "type": "table",
      "page": 3,
      "content": "| 参数 | 说明 | ...",
      "source": "demo.pdf 第 3 页 表格 1"
    }
  ]
}
```

### 4.3 PDF 图片、图注与表格处理

当前 PDF 解析已增强：

- 优先提取 PDF 内嵌图片。
- 对小尺寸空白噪声图片进行过滤。
- 如果论文 PDF 中图是矢量图或页面元素，无法作为普通图片提取，则根据“图1 / Figure 1”等图注定位，裁切图注上方区域，避免整页截图。
- 使用 `pdfplumber` 提取表格，并转换为 Markdown 表格文本供检索使用。
- 报告中表格预览不再显示一串 `|...|` 文本，而是将 Markdown 表格解析为真实网格图片。

关键代码：

```text
parsers/pdf_parser.py
  -> _contains_figure_caption
  -> _caption_crop_bounds
  -> _render_pdf_caption_crops
  -> _extract_pdf_tables

services/report_service.py
  -> _parse_markdown_table
  -> _render_table_preview
```

### 4.4 OCR 与 MLLM 图片理解

入口文件：

```text
services/ocr_service.py
services/llm_service.py
parsers/image_parser.py
```

图片统一进入 `build_image_block`：

```python
local_ocr_text = extract_text_from_image(file_path)
analysis = analyze_image(file_path, local_ocr_text=local_ocr_text)
```

输出内容包括：

- 图片尺寸与格式。
- OCR 文本。
- Qwen-VL 生成的图片说明。
- 可检索的图片内容块。
- 本地 `image_path`，用于报告和问答来源直接展示图片。

## 5. 报告生成流程

入口文件：`services/report_service.py`

报告生成流程：

```text
parsed_documents
  -> 收集 text / image / table 内容块
  -> 表格内容转为语义文本参与总结
  -> 调用 Qwen 文本模型生成摘要和关键信息
  -> 生成 Markdown 报告
  -> 嵌入本地图片 Markdown 引用
  -> 生成表格预览图片
  -> 保存到 output/reports
```

报告结构包括：

- 核心摘要
- 关键信息提取
- 图表/图片说明
- 表格内容摘要
- 待确认或疑问点

前端通过 `_render_markdown_with_local_images` 渲染报告，将本地 Markdown 图片路径转换为 `st.image`，因此报告中的图片、图注裁切图、表格预览图都可以直接显示。

## 6. 手动保存历史会话

当前版本不会自动保存历史会话。

### 6.1 未保存状态

点击“开始分析”后：

```text
上传文件 -> output/temp/unsaved-*
报告文件 -> output/reports/unsaved-*
页面状态 -> st.session_state
SQLite -> 不写入
```

页面会提示：

```text
当前分析仅保存在本次页面状态中，尚未写入 SQLite 历史会话。
```

### 6.2 手动保存状态

用户点击“保存到历史会话”后，系统才会执行：

```python
session_id = create_analysis_session(session_name)
register_uploaded_files(session_id, saved_paths)
save_parsed_documents(session_id, parsed_documents)
save_report_record(session_id, report_path, report_markdown)
```

保存位置：

```text
output/system_state.db
```

SQLite 保存内容包括：

- `analysis_session`：分析会话。
- `document_record`：上传文档。
- `document_block`：文本、图片、表格内容块。
- `report_record`：Markdown 报告。
- `qa_record`：问答记录。
- `agent_run`：Agent 调用轨迹。

只有已保存的会话才会出现在侧边栏历史会话列表中。

## 7. RAG 问答流程

入口文件：

```text
langchain_app/document_builder.py
langchain_app/retrievers.py
langchain_app/chains.py
services/qa_service.py
```

流程：

```text
parsed_documents
  -> build_langchain_documents
  -> Document(page_content, metadata)
  -> BM25 Retriever
  -> FAISS Vector Store Retriever
  -> EnsembleRetriever
  -> Retrieval Chain
  -> Qwen 文本模型生成回答
  -> 来源格式化
  -> 答案校验
```

检索层特点：

- BM25 适合关键词、数字、术语精确召回。
- FAISS + embedding 适合语义相似召回。
- 环境依赖不足时可回退到本地 TF-IDF 检索。
- 图片块也会进入检索，命中图片来源时会返回 `image_path` 和 `image_markdown`。

问答区现在也使用 `_render_markdown_with_local_images`，因此“检索来源”里的图片可以直接显示，而不是只显示图片路径。

## 8. 问题路由与 Workflow

入口文件：

```text
langchain_app/router.py
langchain_app/workflow.py
langchain_app/graph_workflow.py
```

路由器根据中文关键词判断问题类型：

- 文本事实问题：优先 RAG。
- 总结概括问题：优先 RAG。
- 图片理解问题：优先 Agent。
- OCR 问题：优先 Agent。
- 未知问题：优先 Agent。

Workflow 统一调度：

```text
question
  -> route_question
  -> RAG first 或 Agent first
  -> 如果首选链路失败，则回退到另一条链路
  -> 如果 Agent LLM 工具调用失败，则进入确定性工具计划
  -> 输出 answer / sources / steps
```

当前版本已经引入 LangGraph。`answer_with_workflow` 会优先调用 `answer_with_langgraph_workflow`，将路由、RAG、Agent 三个步骤组织成状态图；如果环境缺少 `langgraph` 或图执行异常，则自动回退到原来的函数式 workflow，保证系统可运行。

LangGraph 状态图：

```text
START
  -> route
  -> 条件边：rag 或 agent
  -> rag 节点
      -> 有答案：END
      -> 无答案：agent
  -> agent 节点
      -> 有答案：END
      -> 无答案：rag
  -> END
```

状态对象中保存：

- `question`
- `parsed_documents`
- `session_id`
- `route`
- `result`
- `tried_rag`
- `tried_agent`

## 9. Agent 工具调用流程

入口文件：

```text
langchain_app/agent.py
langchain_app/tools.py
```

当前 Agent 有两层能力：

### 9.1 LangChain Tool-Calling Agent

如果 Qwen Chat 配置可用且 LangChain Agent 正常工作：

```text
question
  -> create_tool_calling_agent
  -> AgentExecutor
  -> 动态调用 tools
  -> final answer + intermediate_steps
```

### 9.2 确定性工具计划兜底

为了减少“Agent 未成功返回结果，已回退到 RAG”，系统增加了确定性工具执行计划：

```text
Agent LLM 不可用 / 工具调用异常 / 返回空
  -> 根据 route intent 选择工具
  -> summary: get_report_summary
  -> image: search_document_blocks(block_type=image) -> analyze_image_with_mllm
  -> ocr: search_document_blocks(block_type=image) -> run_ocr_on_image
  -> text/unknown: search_document_blocks
  -> 返回工具结果和工具轨迹
```

这使 Agent 不完全依赖模型是否一次性正确发起 tool call，提高了稳定性。

## 10. MCP 层设计

入口文件：

```text
mcp_server/main.py
mcp_server/server.py
mcp_server/resources.py
mcp_server/tools.py
langchain_app/mcp_client.py
langchain_app/tools.py
```

当前 MCP 是本地 MCP-style 实现，使用 stdio JSON-lines 通信，已经具备资源和工具两个核心概念。

### 10.1 MCP Resource

当前支持两套 URI：

标准化新 URI：

```text
analysis://sessions/latest
analysis://sessions/{session_id}/report
analysis://sessions/{session_id}/documents
analysis://sessions/{session_id}/qa-history
```

兼容旧 URI：

```text
session://latest
session://{session_id}/report
session://{session_id}/documents
session://{session_id}/qa-history
```

### 10.2 MCP Tool

当前工具：

```text
search_document_blocks
run_image_ocr
analyze_image
get_report_summary
```

工具统一响应结构：

```json
{
  "ok": true,
  "data": {},
  "error": null,
  "metadata": {
    "tool": "search_document_blocks",
    "match_count": 3
  }
}
```

失败时：

```json
{
  "ok": false,
  "data": null,
  "error": "错误信息",
  "metadata": {
    "tool": "analyze_image"
  }
}
```

### 10.3 MCP 与 LangChain Tools 的关系

```text
LangChain Agent
  -> langchain_app/tools.py
  -> StdioMCPClient
  -> mcp_server/main.py
  -> LocalMCPServer
  -> resources / tools
```

如果 stdio MCP Client 启动失败，系统会回退到本地 `LocalMCPServer` 对象直接调用，保证工具层可用。

## 11. 答案校验层

入口文件：

```text
services/answer_validation_service.py
```

问答输出前会执行答案校验：

- 判断是否有来源或工具观测结果支撑。
- 计算答案与证据的关键词重合。
- 检测不确定性表达。
- 对长答案但少证据的情况给出风险提示。
- 输出可信度：高 / 中 / 低。

问答最终输出包括：

```text
路由说明
结果说明
回答
答案校验
检索来源 或 Agent 工具轨迹
```

## 12. 前端展示流程

入口文件：`app.py`

前端主要能力：

- 上传文件。
- 开始分析。
- 报告 Markdown 预览。
- 本地图片直接渲染。
- Markdown 报告下载。
- 手动保存历史会话。
- 加载最近一次或指定历史会话。
- 提问并展示 RAG 来源或 Agent 工具轨迹。
- 问答来源图片直接展示。

本地图片渲染统一由：

```python
_render_markdown_with_local_images(markdown_text)
```

负责。它会识别：

```markdown
![xxx](D:/AgentProject/output/...)
```

并转换为 Streamlit 的图片组件。

## 15. 后续标准化方向

建议继续优化：

1. 引入 `schemas/`，用 Pydantic 定义 `ParsedDocument`、`DocumentBlock`、`QAResult`、`MCPToolResponse`。
2. 新增 `config/settings.py`，集中管理模型、路径、上传限制、检索参数。
3. 把 `app.py` 中的分析主流程抽到 `services/analysis_service.py`。
4. 使用官方 MCP Python SDK 替换当前自定义 JSON-lines MCP-style 协议。
5. 为每个历史会话持久化 FAISS 索引，避免每次问答临时构建。
6. 使用 PaddleOCR 提升中文 OCR 和表格截图识别效果。
7. 继续增强 LangGraph，加入重试策略、人工确认节点和更细粒度的 Agent 状态持久化。
8. 增加端到端测试：上传样例 PDF -> 生成报告 -> RAG 问答 -> 保存历史。
